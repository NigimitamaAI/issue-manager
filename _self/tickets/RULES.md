# プロジェクト運用ルール

## プロジェクト概要
issue_manager — G:\claudedir\ 配下のプロジェクトを横断的にチケット管理するローカル Web アプリ自身の改善タスクを管理する。

## 目的（この運用の）
チケットファイル制により、セッション間の引継ぎをスムーズにし、
トークン消費を抑えながら作業を継続可能にする。

## フォルダ構成
```
tickets/
├── VERSION.md          # issue_manager 対応マーカー
├── RULES.md            # このファイル（運用ルール）
├── INDEX.md            # アクティブチケット一覧
├── TICKET_TEMPLATE.md  # チケット雛形
├── inbox/              # 未整理の入力（アイデア・報告・気付き）
├── todo/               # 精査済み・着手可能
├── doing/              # 着手中
├── blocked/            # ブロック中（外部待ち・判断待ち）
├── done/               # 完了済み
├── archive/            # 退避済み（見えなくするだけ、復元可能）
└── .trash/             # ソフト削除退避先
```

## チケットファイル名規則
形式: `YYYYMMDD_P<優先度>_<工程フェーズ>.md`
例: `20260423_P0_設計.md`, `20260424_P1_実装_API.md`

優先度:
- P0: 最優先（ブロッカー、緊急対応）
- P1: 高（今週中）
- P2: 中（通常タスク）
- P3: 低（余裕があれば）

## ステータス更新ルール（イベント駆動）
- 一会話ごとには更新しない
- inbox → todo: 精査済み（優先度・完了条件・依存関係が届き、セッション開始時に一括整理して昇格）
- todo → doing: セッション開始時に着手チケットを1つ選んで移動
- doing → blocked: 外部待ち・判断待ちが発生したら即移動＋理由追記
- doing → done: 完了条件を全て満たした時点で移動（done 移動時に `- 完了日` が自動追記される）
- セッション終了時: 作業ログに引継ぎメモを1-3行追記

## チケットの日付メタ（起票日・完了日）
- **作成日**: チケット起票時に `- 作成日: YYYY-MM-DD` を記入。TICKET_TEMPLATE.md に組み込み済み。
- **完了日**: done へ移動した時点で自動的に `- 完了日: YYYY-MM-DD` が本文に挿入される（既定であれば上書きなし）
- これらの日付はエクスポート機能でタイムライン管理に活用される

## エクスポート機能
- 画面右上の「⬇ エクスポート」でチケット一覧を以下の形式で出力可能
  - **TSV**: Excel 直開き用（BOM 付き UTF-8、タブ区切り、CRLF）
  - **JSON**: astroritual のタイムラインUI連携用
- 出力列: file / lane / title / priority / status / archivedFrom / createdAt / completedAt / mtime / ctime / path
- 対象レーン: アクティブレーン +（オプションで archive）。.trash は常に除外
- done で完了日が未記入の場合は mtime をフォールバックとして使用

## チケット本体の「ステータス」欄の値
チケット本体先頭の `- ステータス: XXX` に書く値。画面のバッジ表示にも連動する。
- レーン名（`todo` / `doing` / `blocked` / `done`）: 通常はこれ。レーンと一致させる。
- `進行中`: **doing に複数チケットがある時**に、今セッションで実際に作業している方につける。画面に 🟢 緑バッジ。
- `中断`: **doing にあるが別作業を先行中**のチケットにつける。画面に 🟠 オレンジバッジ。中断理由を作業ログに記録。
- `実機検証待ち` / `ユーザー確認待ち` など任意のステータス: 画面にそのままグレーバッジ。blocked に移すほどではないが「状態を見える化」したい場面で使う。

## INBOX 運用
- **アイデア・報告・気付きはすべて inbox/ に投入**
- inbox/ のチケットは以下のいずれかにセッション開始時に振り分ける
  - **todo へ昇格**: 優先度を付け、完了条件を埋め、依存関係を記載
  - **.trash へ廃棄**: 不要または重複と判定
  - **inbox に留保**: 構想段階・状況待ち
- inbox/ のファイル名規則は緩い（簡易名 OK）
  - 例: `IDEA_サーバー冗長化.md`、`BUG_保存時の文字化け.md`
  - todo へ昇格時に `YYYYMMDD_P<優先>_<フェーズ>.md` へリネーム
- **INBOX を使うメリット**: 思いつきを即時記録して doing の本流作業に戻れる

## ARCHIVE 運用
- **見えなくしたいが削除はしたくないチケット**は archive/ へ退避
- 削除と異なり「元のレーンに戻しても文脈を失わない」ように、本文に `- archive元: <元レーン>` を自動記録
- archive/ のチケットは画面上既定で非表示。「archive を表示」ボタンで最右レーンとして表示
- archive のカードには **元レーン名（例: 元: done）のバッジ**が表示される
- **戻し方**: archive のチケットを選択 → 詳細パネルの「元のレーンに戻す」ボタン
- 遡及用途のため archive は原則 Git 管理下に含める（.trash は除外推奨）

## STATUS.md について（自動生成・編集禁止）
- `tickets/STATUS.md` は **サーバーが自動生成**する状態スナップショット
- 生成タイミング: サーバー起動時 + mutation API（チケット作成・編集・移動・削除・archive・restore 等）成功時
- **手で編集しない**。次回 mutation 時に上書きされる
- 含まれる情報: 最終更新時刻 / 各レーン現状 / 直近 done 7件 / 直近 archive 7件 / 統計
- `.gitignore` で除外済み（リポジトリ管理不要、各環境で再生成される）

### INDEX.md との役割分担
- **STATUS.md**: 機械が書く（レーンの実ファイル一覧・統計・直近履歴）
- **INDEX.md**: 人/Claude が書く（推奨実行順・引継ぎメモ・設計判断メモ）

### Claude のセッション開始手順への影響
セッション開始時、Claude はまず `STATUS.md` を読んで実態を把握し、続いて `INDEX.md` で推奨実行順と引継ぎメモを得る。INDEX.md と実態がズレていてもサーバー起動さえすれば STATUS.md は最新になる。

## 作業ログの肥大化対策
- チケット内の作業ログは2層構造にする
  - **サマリ欄**: 要約を記載（Claudeが通常読むのはここ）
  - **詳細ログ欄**: 原文を時系列で全て残す（遡及・監査用）
- 10ターン分程度溜まったらサマリ欄に1行要約を追加
  - 例: 「前半7ターン: Xを試してYに決定」
- **詳細ログ欄の原文は削除・改変しない（追記のみ）**
- セッション開始時はサマリ欄だけ読めば立ち上がれるようにする
- 過去を詳細に遡る必要があるときだけ詳細ログ欄を読む

## チケット分割の粒度
- 1チケット = 1セッションで完結する量を目安
- 大きくなりそうなら最初から子チケットに分ける
- 子チケットのファイル名に親を示唆する語を含める
  - 例: `..._実装_API.md` と `..._実装_DB.md`

## 課題対応の割り込み方針
作業中に緊急度の高い課題が発生したら、以下のいずれかで対応する：

### パターンA: 別チケット化して本流を続行
- 新課題を1件のチケットとして inbox/ または todo/ に起票
- doing の本流作業はそのまま続行
- **使う時**: 本流のゴールが明確で、横道に逸れると戻れなくなる場合

### パターンB: 本流をスタックして課題対応を先行
- doing の本流チケットを一時的に blocked または todo に戻す
- 課題対応チケットを新規作成して doing に
- 本流のチケットにはスタック理由を作業ログに記録
- **使う時**: 課題が本流の前提を壊している（続行しても成果が無駄になるリスクがある）

### パターンC: 本流に組み込む
- doing の本流チケットに完了条件を追加して同セッションで片付ける
- **使う時**: 課題が小さく、独立チケットとするほどではない

### doing は原則 1 つ、例外時は最大 3 つまで
- 複数 doing を許容する場合、それぞれに「進行中/中断」ステータスをチケット本文に明記
- 4 つ以上になったら INDEX.md でレビューして整理

## Claude / 他 AI 向け運用規約 (2026-05-06 改訂 / 2026-05-06 再改訂)

### 重要: Claude の Filesystem MCP には move/delete がない

2026-05-06 一旦「Filesystem MCP に move_file がある」と誘導したが誤り。今の Claude が使える Filesystem MCP は以下のみ:
- read_text_file / read_multiple_files (読み取り)
- write_file / edit_file (書き込み・部分編集)
- create_directory (ディレクトリ作成)
- list_directory / directory_tree / search_files / get_file_info (メタ)

**ファイル移動・ファイル削除・ディレクトリ削除はできない**。よってそれらはファイル操作所バッチ経由でユーザーに依頼し続ける。

### チケットライフサイクル徹底ルール (B 案: 実装セッションでは doing/ 直接書き込み)

**実装を伴うセッションでは、起票時点で直接 doing/ に書き込む** (2026-05-06 確定方針)。

#### ライフサイクルパターン

**A. 実装を伴うセッション (今までの主要パターン)**
  1. Claude: `doing/` 配下に**直接** write_file で起票 (status: doing で記入)
  2. Claude: 実装作業を進める
  3. Claude: 完了時、status を done に edit_file で書き換え、作業ログ/引継ぎメモを更新
  4. Claude: 「このファイルを done/ に移動して下さい」とファイル操作所バッチを生成・ユーザー提示
  5. ユーザー: バッチ実行

**B. 起票だけで終わるセッション (アイデアメモ・将来の TODO 整理)**
  1. Claude: `inbox/` または `todo/` に write_file で起票 (status: todo)
  2. (今回は実装しない) — 次セッションで担当する AI が doing/ に移動

**C. 既存 todo チケットを次セッションで実装する場合**
  1. Claude: status を doing に edit_file で書き換え
  2. Claude: 「このファイルを doing/ に移動して下さい」とバッチ生成・ユーザー提示
  3. ユーザー: バッチ実行
  4. Claude: 実装作業を進める
  5. Claude: 完了時に done 化 (パターン A の 3-5 と同じ)

#### B 案採用の理由
旧方針 (起票時 todo → doing バッチ) は意味区別を保つ代わりに、実装着手前に毎回ユーザーのバッチ実行を待つ必要があった。バッチ実行回数を減らしてユーザーの介入を最小化する方が運用コストが低い。
意味的な区別は緩むが、「inbox/todo にあるもの = 計画/未着手」「doing にあるもの = 着手中」「done にあるもの = 完了」という大枠は維持される。

#### 短時間で終わる作業でも doing/ 直接書き込みを省略しない
たとえ 1 ターンで終わる修正でも、最初から doing/ に書き込む。これはセッション中断時に「今何をしていたか」をユーザーが見れるようにするため。
todo/ に書いて実装を進める運用は禁止。

### Claude 直接実行範囲 (Filesystem MCP の実態)

| 操作 | Claude 直接実行 | ファイル操作所バッチ経由 |
|------|--------------|----------|
| ファイル新規作成 (write_file) | ✅ | ─ |
| ファイル内容部分編集 (edit_file) | ✅ | ─ |
| ディレクトリ作成 (create_directory) | ✅ | ─ |
| ファイル・ディレクトリの移動 | ❌ | ✅ ユーザー実行 |
| ファイル・ディレクトリの削除 (ソフト/物理とも) | ❌ | ✅ ユーザー実行 |
| .bat ファイル生成後の BOM 修復 | ❌ | ✅ BOM修復所_v1.bat 経由 |

「ファイル操作所バッチ」は Claude が「操作指示書 .txt」を生成し、ユーザーが ファイル操作所_v1.bat にドラッグドロップして実行する仕組み。バッチは MKDIR / MOVE / COPY / DELETE / DELETE_PERMANENT / RENAME に対応し、ソフト削除 (.trash への移動) も MOVE で表現できる。

### セッション中断時の状態保全

セッションが途中で切れる可能性を常に考慮する。具体的には:

1. **doing/ にあるチケットには「最終状態」が常に書き込まれていること**
   - 作業ログのサマリ欄に「最後に何をやったか」を 1 行追記
   - 引継ぎメモに「どこで中断したか / 次に何をすべきか」を 1〜3 行で記載

2. **長めの作業中も、節目で作業ログを更新する**
   - 大きな実装ステップが終わるごとに作業ログを追記
   - これにより中断しても次セッションが「3/5 ステップまで終わってる」が分かる

3. **ユーザーから見て「今 doing には何があるか」が真実を反映している**
   - todo/ に置きっぱなしで実装を進めない
   - ユーザーに doing 化バッチを実行してもらってから作業に入る

### 起票専用セッションと作業セッションの区別

- 起票だけで終わるセッション: `inbox/` または `todo/` に write_file (status: todo)
- 起票 + 実装をするセッション: 最初から `doing/` に write_file (status: doing) で起票
- 既存 todo チケットを引き継ぐセッション: 着手前に status: doing に編集 + doing/ への移動バッチを依頼
- どの形でも、**doing/ に置かれたチケット = 「現在 Claude が実装に着手しているもの」** が原則

## セッション開始手順
1. `tickets/RULES.md` と `tickets/INDEX.md` を読む
2. INDEX.md から次に着手するチケットを決定
3. 該当ファイルを todo/ → doing/ へ移動 (status も doing に書き換え)
4. チケット本体を読んで作業開始

## セッション終了手順
1. 作業ログのサマリ欄に今回の成果を1行追記
2. 詳細ログ欄にも今回のやり取りを追記（原文）
3. 引継ぎメモを1-3行で書く
4. 完了していれば done/ へ移動 + INDEX.md 更新
5. 未完了なら doing/ か blocked/ に置いたまま終了

## プロジェクト固有ルール
- issue_manager 自身の改善タスクをここで管理（ドッグフーディング）
- 画面の改善要望は `YYYYMMDD_P<優先>_UI_<機能名>.md` 形式
- サーバーAPIの改善は `YYYYMMDD_P<優先>_API_<エンドポイント名>.md` 形式
- 設計判断が必要なものは `YYYYMMDD_P<優先>_設計_<論点名>.md` 形式

## Windows BAT ファイル編集ルール（最重要）
Claude の `write_file` で `.bat` を生成すると **UTF-8 BOM が付く**ケースがあり、Windows cmd で `'m' は、内部コマンドまたは外部コマンド...` のようなエラーが発生する。

### 必須手順
1. Claude が `.bat` を作成・更新したら、**必ずユーザーへ以下を案内する**：
   > `G:\claudedir\BOM修復所_v1.bat` に修正後の `.bat` をドラッグ&ドロップしてください。UTF-8 BOM を除去し、改行を CRLF に統一し、`.bak` バックアップを作成します。
2. BOM修復所_v1.bat は `logs/` フォルダにログを残すので、問題が出たらそこを確認
3. バックアップ（`.bak`）が自動生成されるので、万一の時はそれに戻せる

### Claude 側の BAT 作成規約
- 原則 **UTF-8 BOMなし**（= ユーザー側で BOM修復所を通してもらう前提）
- `@echo off` を先頭行に、その直後に `chcp 65001 >nul` を置く
- `chcp` より前に日本語（コメント含む）を一切書かない
- 表示メッセージは **英語 + ASCII 中心**（`[OK]` `[ERROR]` `[INFO]` の接頭辞）
- 日本語メッセージを出したい場合は CP932 保存版として別ファイルを用意
- PowerShell 呼び出し時は `[Console]::OutputEncoding=[System.Text.Encoding]::UTF8` を指定

### BAT テンプレート（UTF-8 BOMなし版）
```bat
@echo off
chcp 65001 >nul
setlocal EnableExtensions EnableDelayedExpansion

rem Japanese-safe BAT template
rem Save as UTF-8 without BOM.

set "SCRIPT_DIR=%~dp0"

echo [INFO] Start...

rem main commands

echo [OK] Done.
endlocal
exit /b 0
```

### 禁止事項
- UTF-8 BOM付きで `.bat` を保存する
- 先頭行より前に空白・日本語・BOM を置く
- `chcp` より前に日本語コメントを書く
- 日本語パスや日本語ファイル名の変数を引用符なしで使う

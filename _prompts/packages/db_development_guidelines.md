# DB開発指針

作成日: 2026-06-01
目的: AIがDB投入・移行・大量更新・インデックス作成を提案または実行する前に、負荷を抑えた安全な進め方を提案できるようにする。

## 基本方針

DBに負荷がかかる可能性がある作業では、AIはすぐに実行へ進まず、先に「投入計画」「負荷調査」「停止条件」「検証方法」を提示する。

対象例:

- 大量データ投入
- 既存DBの全件更新
- リレーション/エッジの大量作成
- インデックス/制約の作成・削除
- マイグレーション
- バッチ処理
- Docker内DB、ローカルDB、共有DBへの投入検証

## AIが先に提案すること

AIはDB作業前に、少なくとも以下を提案する。

1. 対象DBと環境
   - ローカル、Docker、共有、検証、本番相当の区別
   - 破壊的操作の有無
   - リセット可能か、既存データを保持すべきか

2. データ量の見積もり
   - 行数、ノード数、リレーション数、ファイルサイズ
   - 最大テーブル/最大ファイル
   - 再実行時に増分か全量か

3. 実行計画
   - 事前準備
   - 事前EXPLAINまたは類似の実行計画確認
   - 本投入
   - 後段インデックス作成
   - 検証
   - ロールバックまたは再生成手順

4. 負荷を下げる工夫
   - アプリケーション経由の大量writeを避け、DBローカルのimport機能を優先する
   - まとめて1巨大トランザクションにしない
   - 適切なバッチサイズを使う
   - endpoint lookupやjoinに必要な最小限の制約/インデックスだけを先に作る
   - 検索/閲覧/デバッグ用の二次インデックスは投入後に作る
   - ネットワーク越しの逐次writeを避ける
   - 再実行可能なスクリプトとmanifestを残す

5. 判断理由
   - 採用方式と不採用方式を比較する
   - 不採用理由を性能、再現性、安全性、運用性の観点で書く
   - 公式ドキュメントやDB固有の推奨がある場合は参照する

6. 実行コマンドと期待結果
   - 後で再利用できるよう、実行コマンドをそのまま記録する
   - 成功時の期待出力、件数、インデックス状態を記録する
   - 停止すべき出力や危険な兆候も記録する

## 実行計画確認の原則

大量投入や大量更新の前には、可能な限りEXPLAIN、QUERY PLAN、DRY RUN、または小規模サンプル実行を行う。

確認する観点:

- 全表スキャン、全ノードスキャンが意図せず発生していないか
- joinやendpoint lookupにインデックスが使われているか
- Eager/materialize系の処理で全件を一度に抱え込まないか
- 1巨大トランザクションになっていないか
- バッチ境界があるか
- ロック範囲が過大でないか

## インデックスと制約の原則

大量投入時のインデックスは「投入に必要なもの」と「投入後に使うもの」を分ける。

事前に作る候補:

- 主キー相当の一意制約
- relationship endpoint lookupやjoinに必要なキー
- 重複防止に必要な最小限の制約

後で作る候補:

- 検索UI用インデックス
- レポート/分析用インデックス
- デバッグ用の補助インデックス
- 投入中に参照しない二次インデックス

判断理由:

- 事前インデックスが少なすぎるとlookupが遅くなる
- 事前インデックスが多すぎると大量write中の保守コストが増える
- 投入後に作成した方が全体として低負荷になる場合が多い

## ネットワーク負荷の原則

大量データは、可能ならDBプロセスの近くに置いてDB自身に読ませる。

推奨:

- DBコンテナ内またはDBが読めるimportディレクトリにCSV/JSON等を配置
- DBのbulk import機能を使う
- Docker volumeまたはbind mountでローカルファイルを読ませる
- manifestで投入件数を管理する

避ける:

- アプリケーションから1件ずつDBへ送る
- ネットワーク越しに大量writeを逐次実行する
- DB投入と無関係なAPI層を通して初期投入する

## ランブック化の原則

DB作業は一度きりのメモではなく、次回も使えるランブックとして残す。

ランブックに含めるもの:

- 前提条件
- 対象環境
- 破壊的操作の有無
- コマンド列
- 期待出力
- 停止条件
- 検証クエリ
- 成功判定
- 再実行時の注意
- 将来切り替えるべき方式

## AI応答用プロンプト

DB作業が含まれる場合、AIは以下の観点をユーザーへ提案する。

```text
DBに負荷がかかる可能性があるため、先に投入計画を作ります。

確認する観点:
- 対象DBと環境
- データ量の見積もり
- 破壊的操作の有無
- 事前EXPLAIN/実行計画確認
- 必要最小限の事前インデックス/制約
- 後段で作成するインデックス
- ネットワーク越しwriteを避ける投入方式
- バッチサイズとトランザクション境界
- 実行コマンド、期待結果、停止条件
- 検証クエリと成功判定

この計画を文書化してから、実行に進みます。
```

## issue_managerへの組み込み

この文書は issue_manager 同梱のパッケージ補助プロンプトとして、`_prompts/packages/` から参照する。
`docs/` は仕様書置き場として扱い、補助プロンプト本文は `_prompts/` 配下に置く。

開発補助プロンプト機能では、以下の仕様を採用する。

- 設定で開発補助プロンプトカタログと選択UIを有効化できる
- 有効化時も、未選択の補助プロンプトはAI向けセッション開始プロンプトまたはチケット読込時プロンプトへ差し込まない
- DB関連語を含むチケットでは、この文書を読むように促す
- 参照する文書パスを設定で拡張できる
- 既存プロジェクト固有RULESを上書きせず、補助文書として追加参照する

設定例:

```json
{
  "assistantPrompts": {
    "enabled": true,
    "enabledPromptIds": [
      "package:db-development-guidelines"
    ],
    "projectPromptIds": [
      "package:db-development-guidelines"
    ],
    "ticketPromptDefaults": {
      "db": ["package:db-development-guidelines"]
    }
  }
}
```

`enabled` は補助プロンプトカタログと選択UIを有効化する。
`enabled` が `true` でも、未選択の補助プロンプトは再開プロンプトへ差し込まない。
`enabledPromptIds` は全体で利用可能な補助プロンプトを制限する。
`projectPromptIds` はプロジェクト再開プロンプトへ既定で差し込む補助プロンプトを指定する。
`ticketPromptDefaults` はトリガー種別ごとのチケット初期選択候補を指定する。
チケット本文に `- assistantPromptIds: package:db-development-guidelines` を書くと、キーワード一致に依存せず明示的に補助プロンプトを差し込める。
新規チケット作成時は、パッケージ補助プロンプトとユーザー定義補助プロンプトをチェックボックスで選択し、選択結果をチケット本文へ保存する。

再開プロンプトへの差し込み順は、共通ルール、拡張機能ルール、補助プロンプト、review/返信確認ルール、対象チケット手順の順とする。

差し込み文例:

```text
開発補助プロンプトがONです。
DB操作、データ投入、マイグレーション、インデックス作成を扱う場合は、先に _prompts/packages/db_development_guidelines.md を読み、負荷を抑えた投入計画・実行計画確認・検証手順を提案してください。
```

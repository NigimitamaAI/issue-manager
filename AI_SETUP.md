# issue_manager AI運用ガイド

このドキュメントは、`issue_manager` を **特定のAI製品に依存せず** 運用するための共通ガイドです。
目的は次の2つです。

- 初回セッションのAIが、どこを読めばよいか迷わないこと
- 既存プロジェクトをチケット運用へ安全に移行できること

`CLAUDE_SETUP.md` は互換性のため残していますが、考え方の本体はこのファイルです。

## 前提

- AIは会話履歴を永続的に信用しない
- プロジェクトの進行状態の正本は `tickets/` 配下のファイルとする
- 特に `STATUS.md` `RULES.md` `INDEX.md` と現在のチケットが、再開時の基準情報になる

## AIが最初に読む順番

対象プロジェクトが既に `tickets/` を持っている場合、AIは次の順で読む。

1. `tickets/STATUS.md`
2. `tickets/RULES.md`
3. `tickets/INDEX.md`
4. `tickets/doing/` のチケット
5. 必要に応じて `tickets/blocked/` `tickets/todo/`
6. そのチケットが参照している仕様書・コード・設定ファイル

原則:
- `STATUS.md` で実在ファイルとレーン状態を確認する
- `INDEX.md` は人やAIが書く要約として扱う
- `INDEX.md` と実ファイルが食い違う場合は、まず `STATUS.md` と実ファイルを優先する

## AIの共通セッション手順

### セッション開始

1. 対象プロジェクトを1つ決める
2. `STATUS.md` `RULES.md` `INDEX.md` を読む
3. `doing/` があれば、その中から今進める1件を選ぶ
4. `doing/` が空なら `todo/` から1件選び、`doing/` へ移す
5. チケット本文の完了条件と引継ぎメモを確認して作業開始

### セッション終了

1. チケットの作業ログサマリを更新
2. 必要なら詳細ログを追記
3. 引継ぎメモを1-3行で更新
4. 完了条件を満たしたら `done/` へ移す
5. 優先順や次アクションが変わったら `INDEX.md` を更新

## 既存プロジェクトを初めてチケット運用へ移行する手順

対象プロジェクトに `tickets/` がまだ無い場合、AIは次の順で移行する。

1. 既存の `README` `docs` `AGENTS.md` `package.json` などを読んで概要を把握する
2. `issue_manager` のテンプレートから `tickets/` 一式を作る
3. `RULES.md` に、そのプロジェクト固有の運用ルールを追記する
4. `INDEX.md` に初期状態を書く
5. 既存の未整理課題を `inbox/` または `todo/` に分解して起票する
6. 最初に進める1件だけを `doing/` に置く

初回移行で最低限必要なチケット:
- 運用開始チケット: このプロジェクトをチケット運用へ移すための作業
- 現状把握チケット: 構成、既知課題、未完了機能の棚卸し
- 最優先実装チケット: 直近で本当に進めるべき開発作業

## AIへの共通プロンプト

以下は任意のAIアシスタントに渡せる共通プロンプトです。

```text
あなたは issue_manager を使う開発アシスタントです。

このプロジェクトでは、会話履歴ではなく tickets/ 配下のファイルを正本として扱います。
まず次の順番で読んでください。

1. tickets/STATUS.md
2. tickets/RULES.md
3. tickets/INDEX.md
4. tickets/doing/ のチケット
5. 必要なら tickets/todo/ と対象コード

ルール:
- doing は原則1件を主対象とする
- 完了条件を満たすまでは勝手に done にしない
- セッション終了時は、作業ログサマリと引継ぎメモを必ず更新する
- INDEX.md と実ファイルがずれていたら、実ファイルと STATUS.md を優先する

もし tickets/ が存在しなければ、まずプロジェクト概要を確認し、チケット運用へ移行するための初期セットアップを行ってください。
```

## 初回運用で起きやすい問題

### 1. AIが広く読みすぎる

対策:
- まず `STATUS.md` `RULES.md` `INDEX.md` に限定する
- 全コード読破を開始条件にしない

### 2. doing が複数あって主対象が曖昧

対策:
- チケット本文の `- ステータス:` を `進行中` / `中断` で分ける
- `INDEX.md` に今回の主対象を明記する

### 3. 会話だけ進んでチケットが更新されない

対策:
- セッション終了前の更新を必須ルールにする
- 少なくともサマリ欄と引継ぎメモは毎回更新する

### 4. 既存プロジェクトの課題が多すぎて最初の1件を選べない

対策:
- まず `現状把握` チケットを1件作る
- そこで棚卸しし、P0/P1だけを `todo/` に昇格する
